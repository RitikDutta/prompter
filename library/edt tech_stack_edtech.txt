We’ll keep the stack practical and aligned with the stage of the product, not overcomplicate it from day one. For the core backend, we’ll use Node.js with a structured framework layer—preferably NestJS if we want better long-term maintainability, otherwise Express if we want to move lighter in the initial phase. PostgreSQL will be the main database because the system is relational by nature—users, courses, chapters, tests, subscriptions, progress, all of that needs clean relations and flexible querying later. Redis will sit alongside it for heavy caching, fast usage-limit checks, dashboard reads, session/token support, and anything we don’t want repeatedly hitting Postgres for. For AI features like the doubt solver, we’ll have a separate RAG flow using embeddings plus a vector store layer—either PGVector if we want to stay closer to Postgres, or a dedicated vector database if scale and retrieval complexity grow. For background and heavier operations like AI processing, notifications, report generation, and delayed jobs, we’ll use a queue layer—RabbitMQ is the better fit if we want strong task-queue semantics, or SQS if we want managed infra on AWS. Docker will be used from the start so every service is containerized and deployment stays consistent across environments. Nginx will sit in front as reverse proxy and load balancer. Kubernetes is not the first milestone, but it becomes relevant when we start horizontally scaling multiple services or handling real traffic spikes cleanly. For admin, React works well because the panel is going to be internal, data-heavy, and iteration-focused. For the mobile app, Flutter gives us a single codebase and fast delivery. So overall, the stack is straightforward: Postgres for source-of-truth data, Redis for speed, backend APIs in Node, queue for async work, AI on a separate retrieval pipeline, Docker for packaging, Nginx for traffic handling, and Kubernetes when scaling actually justifies the operational overhead.
